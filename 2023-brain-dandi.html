Ingredients needed to build an archive

  • People (users are people too!)
  • Standards (define the language)
  • Technologies (they come and distrupt)
  • FOSS (to reuse and share)
  • Automations (people need to sleep)
  • Software needs to speak the languages
  • Technologies they come to bring more data (new data acquisition tech) but also come to store more data, to allow for distributed storage and faster transfer ... so we should always be on lookout
  • Automations are great for all stages in data cycle: more efficient data collection, fewer bugs with data entry, automated verification that all data speaks the same language (standards), that the technologies we deployed performed as they should, that reused components and tested for operating correctly, we to verify that everything speaks the same language

to accommodate growing width (different data types) and size of neural data
  • provides range of interfaces for data contribution and access:
    API, Python CLI client/library, direct AWS S3, FSSPEC, DataLad / git-annex
    WiP: IPFS/FileCoin
  • heavily relies on, and contributes to the development of various standards:
    NWB, BIDS, OME-Zarr, any new one we missed?
  • has modular architecture to reuse and contribute to various FOSS projects:
    (used to: girder,) h5py, hdmf, pynwb, bids-specification, filesystem_spec, git-annex, DataLad, con/tinuous, pyout, ..., yours?
  • integrates with external services:
    MetaCell NWB Exporer, bioimagesuiteweb, Let's integrate with your services!
  • is adopting novel technologies to account for data growth:
    IPFS/Filecoin, Any other we have not heard about?
  • pays special attention to automated software testing and data QC
